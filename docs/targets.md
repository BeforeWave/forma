# Targets

Chinese version: [targets.zh-CN.md](./targets.zh-CN.md)

Forma emits target-specific workflow outputs. A Codex plugin is the Codex
install shape for the same task-level skills. Here, target means the agent
environment that loads the skills, such as Codex, Claude Code, or OpenCode.

The same tracked profile can generate different targets. A workflow generated by
`forma-creator` is fixed to the creator's target. In every path, the output must
match the agent that will load it.

## Supported Targets

| Output | CLI target | Personal install | Project install |
|---|---|---|---|
| Codex skills | `codex` | `$HOME/.codex/skills` | `.agents/skills` |
| OpenCode skills | `opencode` | `$HOME/.config/opencode/skills` | `.opencode/skills` |
| Codex plugins | `codex` | Codex marketplace / plugin UI | Codex marketplace / plugin UI |
| Claude Code skills | `claude-code` | `$HOME/.claude/skills` | `.claude/skills` |
| Claude Code plugins | `claude-code` | `$HOME/.claude/skills/<plugin-name>` | `.claude/skills/<plugin-name>` |

Use the matching `--target` value:

```bash
forma build bundle --target codex --profile <profile.yaml> --output <dir>
forma build bundle --target claude-code --profile <profile.yaml> --output <dir>
forma build bundle --target opencode --profile <profile.yaml> --output <dir>
forma build plugin --target codex --profile <profile.yaml> --output <dir>
forma build plugin --target claude-code --profile <profile.yaml> --output <dir>
```

OpenCode uses the direct skill shape. Forma generates OpenCode-native skill
bundles, not OpenCode JS/TS runtime plugins.

## Codex

Codex reads skills from repository, user, admin, and bundled system locations.

For project skills, Forma installs into `.agents/skills`. User skills install
under `$HOME/.codex/skills`, matching Codex-local skill installs from the Codex
UI and installer surfaces.

For Codex plugins, Forma only generates the local plugin source. Run
`codex plugin marketplace list`, ask the user which existing marketplace to use
or whether to create/register a new one, ensure that marketplace catalog points
to the generated plugin root, then install and enable it through Codex with
`codex plugin add <plugin-id>@<marketplace-name>` or the Codex plugin UI. Start
a new thread after install. If Codex CLI output or marketplace behavior differs,
consult current Codex docs or `codex plugin marketplace --help`.

Run `forma verify <plugin-root>` before handing a plugin to another user or
agent. For structured install handoff, use the `forma build plugin --format
agent|json` output and keep the install route explicit: Codex plugin
installation belongs to Codex rather than `forma install`.

- [Install a local plugin manually](https://developers.openai.com/codex/plugins/build#install-a-local-plugin-manually)
- [Add a marketplace from the CLI](https://developers.openai.com/codex/plugins/build#add-a-marketplace-from-the-cli)

For profile-generated Codex plugins, `<plugin-id>` is the profile `bundle.name`.
For plugin output generated by a Codex `forma-creator`, `<plugin-id>` is
`rename.prefix` when that prefix is present; otherwise it remains `forma`.
`plugin.display_name` sets the Codex plugin install-surface display label
without changing the plugin id, plugin-local skill names, or triggers.
The plugin manifest points to `./skills/`. Forma strips the exact
`<plugin-id>-` prefix from nested plugin skills when present, so plugin `forma`
exposes local stages such as `plan`, `ground`, `lock`, `execute`, and
`showhand`. Renamed profile or creator skills still stay visible through the
plugin after the same prefix rule is applied.

Codex plugin metadata records qualified names such as `forma:plan`. Use
`forma:*` triggers for Codex plugin skills. Direct skill bundles are the shape
that uses standalone `forma-*` skill names.

Codex target bundles may include `agents/openai.yaml` for UI metadata,
invocation policy, and tool dependency declarations.

See the official Codex Agent Skills docs:
<https://developers.openai.com/codex/skills>.

## OpenCode

OpenCode uses direct skill folders. Forma installs OpenCode project skills into
`.opencode/skills` and user skills into `$HOME/.config/opencode/skills`.
OpenCode can also read compatible direct skills from `.agents/skills` and
`.claude/skills`, but Forma's `opencode` install target uses OpenCode's native
skill roots.

Forma does not generate OpenCode JS/TS runtime plugins.

Official docs:
<https://opencode.ai/docs/skills>.

## Claude Code

Claude Code personal skills live under `$HOME/.claude/skills`. Project skills
live under `.claude/skills`.

Claude Code also discovers project skills from parent directories and nested
`.claude/skills` directories while working in a repository. Project skills may
require workspace trust before tool permissions apply.

For Claude Code plugins, Forma generates a plugin root with
`.claude-plugin/plugin.json`, root `.forma-manifest.json`, and nested
`skills/<skill-name>/` directories. Verify it, then install the plugin root with
`forma install --target claude-code --scope user|project <plugin-root>`.

Claude Code plugin skills are plugin-local. When a generated plugin skill name
starts with the exact plugin name plus `-`, Forma strips that prefix in plugin
output. For plugin `forma`, the plugin-local stages are `plan`, `ground`,
`lock`, `execute`, and `showhand`. Direct skill bundles use the `forma-*`
standalone skill-name pattern instead.

See the official Claude Code skills docs:
<https://code.claude.com/docs/en/skills>.

## Target-Specific Output

Target adapters affect emitted metadata and install behavior. They should not
change the task-level workflow contract itself.

For example:

- Codex output may include `agents/openai.yaml`.
- Claude Code output should not contain Codex-only metadata.
- OpenCode output uses native direct skill frontmatter with
  `compatibility: opencode` and string-to-string `metadata`; it should not
  contain `agents/openai.yaml`, `.codex-plugin`, or `.claude-plugin` metadata.
- Codex plugin output includes `.codex-plugin/plugin.json`, root
  `.forma-manifest.json`, and nested `skills/`.
- Codex plugin `plugin.json` uses the profile `bundle.name` or creator
  `rename.prefix` as plugin id, and its `skills` path must resolve to the
  nested `skills/` directory.
- Claude Code plugin output includes `.claude-plugin/plugin.json`, root
  `.forma-manifest.json`, and nested `skills/`.
- Claude Code plugin `plugin.json` uses the profile `bundle.name` or creator
  `rename.prefix` as plugin name, and its `skills` path must resolve to the
  nested `skills/` directory.
- Skill directory names and frontmatter must match the target-readable bundle
  contract.

## Distribution Boundary

Direct skill directories are good for local trial use and project-scoped
workflows.

If a workflow should be distributed more broadly, package or distribution
mechanisms may become target-specific. In the durable path, a Forma profile is
the workflow source and target outputs are generated from it. In the creator
path, try the generated output first, then promote useful rules into a profile.

## Review Before Trust

Generated bundles can include scripts. Review project bundles before trusting
them, especially when a target allows tool permissions or dynamic context
injection.
