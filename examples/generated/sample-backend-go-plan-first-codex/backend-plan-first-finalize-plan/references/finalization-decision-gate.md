- Before reading planning references, running `scripts/issue-workflow.sh init <issue-id>`, or writing `plan.md` / `tasks.md`, confirm from the current user-agent conversation alone that Goal, Scope, Approach, Validation, Plan Strategy, selected grounding producer or confirmed grounding handoff, and any applicable Artifact/Evidence Boundary are already decision-complete. Fail closed: if any unanswered question could still change the deliverable, module scope, implementation shape, artifact/evidence boundary, or acceptance criteria, stop and tell the user what still needs to be clarified instead of writing a partial plan.
- When blocked, respond in the compact `blocked` format from `references/output-format.md`: one sentence, a `Missing:` line naming only unconverged dimensions, and up to 3 short clarifying questions.
- Do not infer that a task is documentation-only, analysis-only, or safe for `# no-programmatic-validation: <reason>` unless the current conversation explicitly settles that no code behavior, code-owned definitions, or runtime logic needs to change.
- Treat Goal as unconverged unless the current conversation explicitly settles the concrete deliverable to produce.
- Treat Scope as unconverged unless the current conversation explicitly settles both what is included and what is excluded.
- Treat Approach as unconverged unless the current conversation explicitly settles the intended deliverable shape, the concrete surfaces to touch, and whether the work adds new assets or edits existing ones. Broad prompts such as "put a markdown file under auth", "add a markdown doc in the auth module", or "write it under that directory" are still blocked if the agent would need to choose the actual file path, file count, or touched files.
- Treat Validation as unconverged unless the current conversation explicitly settles each task's completion contract, any reusable task-safe shared checks, and the issue-level final validation or explicit review-only acceptance standard.
- Treat Plan Strategy as unconverged for new plans unless the conversation explicitly selects `step-execution`, `loop-exploration`, or `hybrid`; existing legacy plans without this field default to `step-execution`.
- Treat grounding as unconverged unless the current conversation includes a selected grounding producer still to be run, or a confirmed grounding handoff whose known facts are sufficient for the final plan.
- For `loop-exploration` or `hybrid`, do not finalize until baseline metric, target metric or convergence threshold, batch selection, batch size, artifact path/type, stop/skip/retry rules, no-full-rerun guard, formal/destructive write boundary, and final validation are settled.
- Treat the Artifact/Evidence Boundary as unconverged for `loop-exploration`, `hybrid`, generated-artifact, import/export, migration, batch-processing, formal/destructive write, or evidence-producing plans unless the current conversation explicitly settles artifact paths, evidence paths, committed/gitignored/transient output policy, source-of-truth status, state or environment assumptions, proof gates, metadata/provenance requirements, write boundaries, and final validation.
- When multiple requirement, design, contract, solution, grounding, or conversation inputs are present, treat `Source-of-truth refs` as unconverged unless the current conversation states the precedence of the sources actually present. Do not add a generic list of absent source categories.
- Do not downgrade a user-required proof into a weaker surrogate, postpone it to post-commit, or choose a report/evidence path during finalization unless that boundary was already explicit in the conversation.
- If the final plan would require choosing whether local outputs are committed, gitignored, transient, or source-of-truth, stay blocked instead of writing a partial plan.
- If repository exploration would still need to choose a concrete file path or target file, create-versus-edit behavior, single-versus-multi-file output, touched interface, validation mode, or source precedence, stay in clarification mode and do not start plan writing. `finalize-plan` must not run repository exploration to fill those gaps.
